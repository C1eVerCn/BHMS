# BHMS 文档分析报告

更新日期：2026-03-15

## 1. 结论

当前仓库的主要问题不再是“代码没有实现”，而是文档曾经存在明显的口径分裂：

- 一部分文档把系统写成早期 MVP
- 一部分文档把目标态写成已完成
- README、页面和代码能力一度没有完全对齐

本轮收尾后的文档治理原则已经固定为：

1. `README.md` 负责说明当前真实能力
2. `Doc/系统使用说明.md` 负责启动与操作
3. `Doc/实验复现说明.md` 负责实验与案例资产复现
4. `Doc/答辩演示手册.md` 负责现场演示口径
5. `Doc/BHMS系统设计文档_整合版.md`、`Doc/BHMS系统开发文档_整合版.md` 仅保留为参考 / 目标态文档

## 2. 当前代码与文档对齐情况

### 已对齐部分

- 多 seed 与 ablation 已明确写成已完成能力
- 案例目录导出已明确写成已完成能力
- GraphRAG explainability 已明确写成已完成能力
- `data/processed` 默认基线与本地训练池导出已经区分清楚

### 当前仍需继续补强的真实问题

- 多个来源实验指标仍偏弱
- `data/models/**` 与部分案例资产里仍有绝对路径
- 一键复现命令链仍需进一步封装

## 3. 风险判断

### 低风险

- README、使用说明、答辩手册彼此冲突
- 演示样本污染仓库默认 `data/processed`
- 把“脚本已存在”写成“脚本未完成”

### 中风险

- 论文中如果直接引用当前实验结果，仍可能因为 `R²` 偏弱被追问
- `data/models/**` 中的绝对路径仍会影响换机复现

## 4. 建议的后续治理顺序

1. 先清理 `data/models/**` 的绝对路径
2. 再优化实验质量与论文结论
3. 最后补最小一键封版脚本

## 5. 文档使用建议

### 面向老师 / 评审

优先看：

- `README.md`
- `Doc/系统使用说明.md`
- `Doc/答辩演示手册.md`

### 面向开发者

优先看：

- `README.md`
- `Doc/实验复现说明.md`
- `frontend/docs/ui-guidelines.md`

### 面向后续扩展

参考看：

- `Doc/BHMS系统设计文档_整合版.md`
- `Doc/BHMS系统开发文档_整合版.md`

## 6. 总结

当前 BHMS 仓库已经具备比较完整的毕业设计成品骨架。文档治理的关键任务已经从“补文档”转变为“确保任何文档都不夸大、不落后、且与仓库真实状态一致”。
