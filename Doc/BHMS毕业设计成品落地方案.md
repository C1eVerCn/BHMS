# BHMS 毕业设计成品落地方案

更新日期：2026-03-17

## 文档定位

本文件不再承担长期路线规划职责，只负责描述：

- 当前仓库已经完成到什么程度
- 现阶段最需要封版的内容是什么
- 在答辩和阶段性交付前，应优先补哪些短板

长期主线、阶段计划与后续实施路线统一以 `Doc/BHMS统一框架实施计划.md` 为准。

## 当前仓库状态判断

截至 2026-03-17，仓库已经具备以下基线能力：

- 三源数据导入与基础训练闭环已跑通：`NASA / CALCE / Kaggle`
- `Bi-LSTM` 与 `Hybrid` 的训练、multi-seed 汇总、ablation 基础框架已具备
- 推理、异常检测、GraphRAG 诊断、案例导出和前端展示已形成可演示链路
- 近期已开始向 `xLSTM` 优化版 Hybrid 与实验资产收口过渡

因此，项目的真实状态不是“缺一个完整系统”，而是“已有一套能跑的系统，需要升级成更强、更清晰、更可复现的统一框架”。

## 当前封版优先级

### 1. 仓库结构收口

优先保证：

- 主文档集合清晰
- 过时脚本和误生成目录移除
- 实验资产目录不再混杂搜索残留与临时优化目录
- `README`、文档和仓库结构口径一致

### 2. 实验资产可复现

优先保证：

- `data/models/**` 的 summary / metadata / checkpoint 引用改成仓库相对路径
- 图表 metadata 和 plot manifest 不再写入机器绝对路径
- 新机器克隆后能够直接读取现有实验资产

### 3. 生命周期路线切换

后续封版不再继续强调“单点 RUL 原型”，而是切换到：

- `xLSTM + Transformer` 生命周期混合模型
- 多源生命周期 benchmark
- `GraphRAG` 未来退化机理解释

### 4. 论文与答辩材料同步

论文暂不急于定稿，但封版内容必须逐步形成：

- 生命周期预测图
- `knee / EOL / RUL` 联合输出图
- GraphRAG 证据链
- 标准案例导出目录
- 一套可复述的实验对照口径

## 当前主要短板

### 科研有效性

- 当前主结果仍以 `RUL` 回归为主，生命周期任务尚未成为仓库默认主线
- `Hybrid` 相对 baseline 的稳定优势尚未在所有来源上固化
- 多源迁移和更大规模数据集尚未真正接入

### 可解释性深度

- GraphRAG 当前更偏“当前异常诊断”
- 未来退化阶段、风险窗口和候选机理排序仍需升级
- 模型输出与知识图谱之间的 lifecycle 证据接口尚未完成统一

### 数据规模

- 目前可直接用于主实验的数据量仍偏小
- `NASA + CALCE` 难以支撑“全生命周期统一框架”的说服力
- `HUST / MATR / Oxford / PulseBat` 接入后，结论可信度才会明显上升

## 封版阶段的工作原则

- 不再新增重复规划文档，所有长期计划统一维护在主计划文档中
- 不以页面堆叠为优先，而以数据、模型、解释链三条主线为优先
- 不转向纯 Transformer 路线，坚持 `xLSTM` 为生命周期记忆主干
- 不让外部 LLM 介入机理排序，只允许其做解释摘要增强

## 当前结论

BHMS 现阶段已经有毕业设计成品骨架，但下一轮真正决定项目质量的，不是继续做更多零散功能，而是完成以下三件事：

1. 把仓库整理成清晰、可复现、可提交的基线
2. 把数据与任务升级为多源全生命周期预测
3. 把 GraphRAG 升级为面向未来退化机理的解释系统

完成以上三件事后，项目才算真正进入“统一框架封版”阶段。
