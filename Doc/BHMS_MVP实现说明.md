# BHMS MVP 实现说明

本文档用于说明当前仓库代码与 `Doc` 目录中整合设计/开发文档之间的关系，避免“文档是目标态、代码是样机”带来的理解偏差。

## 1. 当前代码定位

当前代码实现的是“毕业设计 MVP”，目标是：

- 使用真实 NASA 电池数据跑通最小可演示闭环
- 保证前后端联调后可直接展示预测与诊断结果
- 为答辩、论文截图、实验流程说明提供稳定工程底座

不追求一次性完成完整工业化能力。

## 2. 已经实现的核心闭环

- NASA MAT 原始数据 -> 周期级结构化 CSV
- 周期级数据 -> SQLite 入库
- SQLite 数据 -> 电池列表 / 健康状态 / 周期曲线
- 周期序列 -> RUL 预测（训练模型或启发式 fallback）
- 周期指标 -> 异常检测 -> 标准化症状事件
- 标准化症状事件 -> 本地知识库匹配 -> 诊断报告生成
- React 前端 -> 真实 API 展示

## 3. 与整合设计文档的差异

### 已对齐的内容

- 修正了 xLSTM 不应宣称“线性计算复杂度”的表述边界
- 增加了异常检测的明确输出接口
- 增加了分页电池列表与历史记录持久化
- 增加了训练脚本、数据预处理脚本与本地知识库文件

### 有意延后的内容

以下内容仍保留在目标态文档中，但未纳入本轮 MVP：

- JWT 认证与权限系统
- Redis / Celery 异步任务体系
- Prometheus / Grafana 监控
- Docker / Docker Compose 部署编排
- Neo4j 生产化图数据库接入
- 限流、中间件安全加固、CI/CD 工作流

## 4. GraphRAG 的 MVP 约束

当前 GraphRAG 不是完整的“图数据库 + 嵌入检索 + 在线 LLM 推理”方案，而是：

- 以 `data/knowledge/battery_fault_knowledge.json` 作为本地知识图谱近似载体
- 基于症状匹配、规则评分和模板化语言生成输出诊断结果
- 保留 `KnowledgeGraph` / `LLMInterface` / `GraphRAGEngine` 三层边界，便于后续切换到 Neo4j 和真实 LLM API

## 5. 建议的答辩表述

建议在答辩或论文中使用以下表述：

> 本系统当前完成了以 NASA 公开数据集为基础的 BHMS MVP，实现了从原始数据导入、RUL 预测、异常检测到基于知识库的诊断报告生成的完整演示闭环；工业化部署与更复杂的知识图谱能力作为后续扩展方向保留。
